🤖 Role: Senior Python Backend Architect
Expertise: FastAPI, modular refactoring, asynchronous task processing (Email/OCR), and system maintainability.

🎯 Task: Structural Refactoring of fapiaohezi (Backend: fp_api)
Core Objective: Transition from a "technically over-layered" structure (too many services/utils/helpers) to a "Business-Feature Centric" modular structure.
Key Goal: Ensure one feature is contained within one file, and one API's main logic is visible within one primary function.

1. Refactoring Constraints (The "Never" List)
NO over-splitting: Stop jumping between 4+ files for a single API.

NO redundant abstraction: Eliminate unnecessary Service, Helper, and Wrapper layers.

NO breaking changes: Maintain existing API endpoints and business logic.

NO "Ghost" logic: Route files must contain logic, not just forward requests to services.

2. New Target Architecture (Flat & Feature-Based)
Plaintext
fp_api/
├── main.py          # App entry & middleware
├── config.py        # Global settings
├── database.py      # DB connection & Session
├── models.py        # Data models (keep unified or minimal)
├── auth_api.py      # ALL Auth logic: Register, Login, Profile
├── invoice_api.py   # ALL Invoice logic: Upload, Batch, OCR Submit, Query
├── email_api.py     # ALL Email logic: Pull, Filter, Parse, Attachments
└── response.py      # Unified response schemas
3. Atomic Logic Requirements (Feature Breakdown)
A. Auth Module (auth_api.py)
Consolidate: Register, Login, SMS (placeholder), and Profile management.

Structure: One function per endpoint. Linear logic: Validate -> DB Query -> Verify -> Return.

B. Invoice Module (invoice_api.py) - CRITICAL
Function 1: upload_invoices_and_create_batch

Steps: Receive files -> Get user_id -> Generate batch_id -> Validate size/format -> Save disk -> DB Write -> Return.

Function 2: start_batch_recognition

Steps: Fetch files by batch_id -> Format OCR params -> Remote API Call -> Update status -> Return.

Function 3: query_batch_status

Steps: DB Query -> Aggregate by filename -> Return.

C. Email Module (email_api.py)
Consolidate: Connection, INBOX selection, Date filtering, PDF parsing, Batch creation, and OCR submission.

Structure: Maintain a single "Main Flow" function that tracks the process from login to DB entry.

4. Implementation Protocol
Scan & Audit: Identify specific files/functions that are currently over-split.

Structural Mapping: Propose the exact file merges.

Sequential Refactor:

Phase 1: auth (Simple)

Phase 2: invoice (Heavy logic)

Phase 3: email (External integration)

Clean up: Delete or merge the legacy services/, utils/, and helpers/ folders.

5. Definition of Done (DoD)
Can I understand the full flow of "Upload to Recognition" by reading one file?

Are cross-file references reduced by at least 50%?

Is the "Batch Creation" logic explicitly a single, step-by-step function?

Does the project remain locally runnable with no changed endpoints?
